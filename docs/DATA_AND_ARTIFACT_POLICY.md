# Data and Artifact Policy — Sample Brain

## 1. Purpose

This document defines what data and artifacts may be committed to version control, what must stay local, and how to handle generated or downloaded files. It applies to every contributor, automation script, and documentation example in this repository.

**No commit shall introduce generated state, private data, or sample audio into the Git history.** If it can be regenerated, it must not be committed.

---

| Category | In Git | Always untracked | Rebuildable |

| `reports/` | ❌ No | ✅ Always | ✅ Yes |
| `data/indexes/`, `*.faiss`, `*.index` | ❌ No | ✅ Always | ✅ Yes (from `sample_embeddings`) |

## 3. Source of Truth (Committed)

The following **must** be committed. They define the system and its behaviour:

| Artifact | Location | Why |
|----------|----------|-----|
| Source code | `src/*.py` | All pipeline modules, CLI, config, DB helpers |
| Package metadata | `pyproject.toml` | Project name, entry point, build system |
| Python dependencies | `requirements.txt` | Pinned runtime dependencies |
| Python version | `.python-version` | Deterministic Python version |
| Documentation | `docs/` | Product requirements, system requirements, target architecture, artifact policy |
| Architecture records | `knowledge/roadmap/adr/` | Design decisions with rationale |
| Roadmap and status | `knowledge/ACTIVE_ROADMAP.md`, `knowledge/CURRENT_STATUS.md` | Current and planned work |
| Seed labels | `data/label_seeds.csv` | Weak labels for kNN autotype (small, manually curated) |
| Regex maps | `data/filename_tag_regex.json` | Filename-based tag inference (manually curated) |
| CI/CD config | `.github/workflows/` | CI/CD configuration and repository automation definitions |

### 3.1 Rules for tracked data

- Seed files and regex maps must be **small, manually curated, and versioned intentionally**.
- Any data file over 100 KB in `data/` must be reviewed before commit.
- No file containing absolute paths, usernames, or environment-specific values shall be committed.

## 4. Local Runtime State (Never Committed)

The following may exist on the local filesystem but must never appear in `git status` or a commit:

| Artifact | Location | Content | Rebuildable |
|----------|----------|---------|-------------|
| SQLite database | `data/catalog.db` | All sample metadata, features, embeddings | ✅ Yes — `init` → `scan` → `analyze` → `autotype` |
| Last scan state | `data/last_paths.json` | Tracks previously scanned roots | ✅ Yes — deleted and recreated on next scan |
| Reports | `reports/` | CSV exports, validation reports, title suggestions | ✅ Yes — regenerated by pipeline scripts |
| Vector indexes | `data/indexes/` | NumPy `.npz` index files (optional external cache) | ✅ Yes — `sample-brain index_build --save` |
| sqlite-vec vec0 cache | Inside SQLite DB (`vector_index_state`, vec virtual table) | Rebuildable ANN cache in catalog DB | ✅ Yes — `index_build --search-backend sqlite-vec` |
| Embedding storage | `data/embeddings/` | NumPy files, embedding caches | ✅ Yes — regenerated by `sample-brain embed` |
| Model weights | `data/models/` | Downloaded CLAP/OpenL3 weights (`.pt`, `.pth`, `.safetensors`) | ❌ — Downloaded one-time; cache in `.gitignore` |
| Hugging Face cache | `~/.cache/huggingface/` | Model weights downloaded by transformers | ❌ — Outside repo entirely |
| Virtual environment | `.venv/` | Installed Python packages | ✅ Yes — `python -m venv .venv && pip install -r requirements.txt` |
| Python cache | `__pycache__/`, `*.pyc` | Bytecode cache | ✅ Yes — never committed |
| FL Studio tags | User-specified path (e.g. `Documents/Image-Line/Settings/Browser/Tags`) | Exported browser tags | ✅ Yes — `sample-brain export_fl` |
| Agent session state | `knowledge/SHARED.WORKING.MEMORY.md`, `knowledge/logs/` | Private working memory and logs | N/A — explicitly excluded |

### 4.1 Rules for local state

- **Never** `git add` any file matching patterns in Section 7, even if it appears in `git status`.
- If a generated artifact must be shared for debugging, copy it outside the repository and share manually.
- If the SQLite database or an index file is accidentally staged, stop and unstage it before committing.
- The Hugging Face cache is system-global. Never copy it into the repository.

## 5. Generated Artifacts vs One-time Downloads

### 5.1 Rebuildable artifacts

The following can always be regenerated from committed source code and local sample data:

```text
data/catalog.db          →  rebuild from scan → analyze → autotype pipeline
reports/*                →  project-specific report generation commands, if present
data/indexes/*.npz        →  rebuild from stored sample embeddings (current NumPy format)
data/indexes/*.faiss     →  N/A (superseded by sqlite-vec / NumPy)
data/embeddings/*.npy    →  rebuild from the embedding pipeline, when file-based embedding caches are used
.venv/                   →  python -m venv .venv && pip install -r requirements.txt
```

### 5.2 One-time downloads (not rebuildable from source)

The following are downloaded from external sources and cached locally. They are not in the repository but are required for optional functionality:

| Artifact | Source | Size | Triggered by |
|----------|--------|------|-------------|
| CLAP model weights | Hugging Face (`laion/clap-htsat-unfused`) | ~500 MB | First call to `get_backend("clap").embed_audio()` |
| torch + transformers | PyPI (via optional `[clap]` extra) | ~2-3 GB | `pip install sample-brain[clap]` or manual install |

**Policy:** These downloads must be optional, user-initiated, and never happen during `pip install sample-brain` or `sample-brain --help`.

## 6. Boundary Cases

### 6.1 Audio sample files

No audio files (`.wav`, `.aiff`, `.flac`, `.mp3`, `.ogg`, `.aif`) are ever committed. The repository contains only code, metadata, and documentation. Samples are analysed in place on the producer's local filesystem.

**Exception:** If a minimal synthetic test fixture is needed, prefer generating it during tests rather than committing binary audio. If a committed fixture is unavoidable:
- It must be synthesised by a script (not committed as a binary blob)
- It must be under 50 KB
- It must live in a dedicated directory such as `tests/fixtures/`
- It must be explicitly listed in `.gitignore` exceptions if absolutely necessary

### 6.2 Configuration files with local paths

`src/config.py` currently contains hardcoded `SAMPLE_ROOTS`. This is known technical debt (see TARGET_ARCHITECTURE.md Section 2.4). Until EPIC 1 is implemented:

- The committed version of `config.py` must contain a **placeholder or well-documented default**, not the actual local path of the developer who last committed.
- Use a placeholder such as `<SAMPLE_LIBRARY_ROOT>` in documentation examples, or read the path from an environment variable or local-only config file.
- Future EPIC 1 will introduce a proper profile-based configuration system.

### 6.3 Reports and temporary files

The `reports/` directory is fully untracked. If a report is worth sharing (e.g., a validation benchmark), export it outside the repository or inline the relevant data into documentation.

### 6.4 Agent/IDE state files

Agent session logs, working memory, IDE metadata (`.idea/`, `.vscode/`), and cursor indexing configurations are untracked. They are runtime state, not project artifacts.

## 7. Gitignore Reference

The following patterns are covered by `.gitignore` (see the file itself for the full list):

| Pattern | What it excludes |
|---------|------------------|
| `.venv/` | Virtual environment |
| `__pycache__/` | Python bytecode cache |
| `data/catalog.db` | SQLite database |
| `data/**/catalog.db` | SQLite database in subdirectories |
| `data/last_paths.json` | Last scan state |
| `*.db`, `*.sqlite`, `*.sqlite3` | Any database files |
| `data/indexes/`, `data/embeddings/` | Vector index and embedding files |
| `*.faiss`, `*.index` | Legacy FAISS index files (superseded by ADR-0004) |
| `*.npy`, `*.npz` | NumPy array files |
| `*.pkl` | Pickle files |
| `data/models/` | Downloaded model weights |
| `*.pt`, `*.pth`, `*.safetensors` | PyTorch / ML model weights |
| `reports/` | Generated reports |
| `data/**/*.wav`, `data/**/*.aiff`, etc. | Audio sample files |
| `knowledge/SHARED.WORKING.MEMORY.md` | Private working memory |
| `knowledge/logs/` | Agent session logs |

### 7.1 Keeping gitignore up to date

If a new command or pipeline step introduces a new generated artifact type:

1. Add the corresponding pattern to `.gitignore`
2. Add the artifact to Section 4 of this document
3. Verify with `git status --short` that no unintended files appear

## 8. Enforcement

### 8.1 Per-commit checklist

Before every commit, verify:

- [ ] `git status` shows only intended files (no `.db`, `.faiss`, `.npy`, `.wav`, `.venv/`, `reports/`, `__pycache__/`)
- [ ] No files containing absolute local paths or personal data
- [ ] Documentation does not reference private or environment-specific paths
- [ ] If a new file type is being committed, it is documented in this policy

### 8.2 If an artifact is accidentally committed

1. **Stop.** Do not push.
2. Remove the file: `git rm --cached <file>` (keeps it on disk, removes from tracking).
3. Add the pattern to `.gitignore` if it is missing.
4. Amend the commit: `git commit --amend` (if unpushed) or create a new commit.
5. Verify: `git status` is clean and the file is no longer tracked.

### 8.3 Automation

The CI smoke workflow (if running) checks that `git status --short` is clean after all pipeline tests. A dirty status at the end of CI indicates that generated artifacts were produced and not cleaned up — this is a test failure.
